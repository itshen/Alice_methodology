# 第十三章：可观测性

> 不可观测的系统等于黑盒；黑盒不能优化，也不能被信任。

---

## Agent 的可观测性挑战

普通 Web 应用的可观测性相对简单：请求进来，处理，响应出去，记录请求日志。

Agent 系统有独特的挑战：

**非线性执行**：多轮迭代（每轮可能调用多个工具）、可能递归的子 Agent、后台的记忆提取和 Skill 分析交织在一起。

**时间跨度大**：一次任务可能持续几分钟甚至更长。中间发生了什么，如果没有记录，事后几乎无法还原。

**原因难以归咎**：出了问题（Agent 做了意外的操作、任务没有完成），是工具描述误导了模型？还是上下文压缩丢失了关键信息？还是权限规则阻止了正确的操作？

**隐私敏感**：对话内容可能包含敏感信息，不能无差别记录所有内容。

## OTel 三信号

OpenTelemetry 定义了三种可观测信号，都适用于 Agent：

### Traces（追踪）

记录一次完整操作的调用链：

```
interaction（一次用户对话）
│
├── llm_request（LLM API 调用）
│   ├── model: qwen3.6-plus
│   ├── prompt_tokens: 3200
│   └── completion_tokens: 450
│
├── tool_execution（工具执行）
│   ├── tool_name: web_search
│   ├── blocked_on_user: 0ms   ← 等待用户确认的时间
│   └── execution: 1200ms      ← 实际执行时间
│
├── compress（压缩事件）
│   ├── level: micro_compact
│   ├── tokens_before: 85000
│   └── tokens_after: 62000
│
└── subagent（子 Agent）
    └── （子 Agent 的完整 trace，嵌套）
```

**`blocked_on_user` vs `execution` 的分离**：这是一个重要的设计细节。如果不区分这两段时间，你会在日志里看到工具执行"花了 30 秒"，但实际上其中 28 秒是在等用户点击确认按钮。分开记录才能准确分析系统性能。

### Metrics（指标）

关键指标：
- LLM 请求总数 / prompt tokens 总量 / completion tokens 总量
- 工具调用次数（按工具分类）
- 压缩触发次数（按压缩级别分类）
- 会话持续时长分布

### Logs（结构化日志）

每个关键事件的结构化记录，按日期落盘，支持归档和分析。

## LLM 调用分类

每次 LLM 调用都需要知道它是为什么目的发起的：

```
caller 字段的可能值：

main        主对话（用户直接触发）
memory      记忆提取（对话结束后的后台任务）
compact     上下文压缩
classifier  权限分类器（dont_ask 模式下）
subagent    子 Agent 的调用
```

这个分类让你能回答：
- "为什么这次对话花了这么多钱？" → 看 caller 分布
- "压缩操作频繁吗？" → 看 compact 类型的调用
- "子 Agent 消耗了多少 token？" → 看 subagent 类型的调用

## 隐私保护默认

可观测性和隐私保护之间有天然张力：记录越详细，越容易调试；但记录越详细，用户隐私暴露越多。

**Alice 的原则**：隐私保护是默认值，详细记录是选择项。

默认情况下：
- 不记录 Prompt 内容
- 不记录工具的具体输入参数
- 只记录元数据（时间、模型、token 数、工具名、耗时）

需要调试时，用户显式开启详细日志模式，记录完整的请求/响应。

这和权限系统的设计哲学一致：**默认保守，显式放宽**。

## 启动性能追踪

Agent 应用的启动速度影响用户体验。在关键初始化阶段打点：

```
进程启动
    ↓ mark("imports_done")     — 模块加载完成
    ↓ mark("storage_ready")   — 数据库初始化完成
    ↓ mark("mcp_ready")       — MCP 连接就绪（可选）
    ↓ mark("window_shown")    — 用户看到界面
```

每个 mark 记录相对进程启动的耗时（毫秒）。

这类数据帮助识别：哪个初始化步骤最慢？某次版本更新后启动变慢了，是因为什么？

## 多 Agent 的可观测性

单个 Agent 出问题好定位；多个 Agent 并发时，日志混在一起很难区分。

解决方案：**每个 Agent 实例有唯一 ID**，所有日志和事件都携带这个 ID。

在 OTel Trace 里，子 Agent 的 Span 是父 Agent Span 的子节点，形成完整的树状调用链。在 UI 里，子 Agent 的事件通过 `subagentId` 字段对应展示在各自的卡片里。

## PlayGround 的可观测性调试工具

可观测性不只是线上监控，也是开发时的调试工具。

PlayGround 是专门的调试面板，包含：
- 实时查看当前上下文大小和压缩状态
- 模拟不同的权限模式，观察系统行为
- 查看 MCP 连接状态
- 测试不同的提示词，观察模型响应

在开发新功能时，PlayGround 是"在不改生产代码的情况下验证想法"的地方。

## 可观测性的工程价值

可观测性不是运维需求，是工程质量的标志。

**没有可观测性的系统的典型表现**：
- 出了问题只能靠重现和猜测
- 性能优化无从下手（不知道瓶颈在哪）
- 用户报告"偶发问题"无法排查

**有可观测性的系统**：
- 问题出现时，日志告诉你发生了什么，为什么
- 可以用数据驱动优化（某个工具被调用太频繁、压缩触发太早）
- 监控关键指标，提前发现异常

对于 Agent 系统，可观测性尤其重要。执行路径太复杂，人工推理几乎不可能，**只有靠数据才能理解系统的行为**。
