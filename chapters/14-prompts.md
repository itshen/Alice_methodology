# 第十四章：提示词工程

> System Prompt 不是一段文字，是精心设计的分层注入系统。

---

## 提示词的分层

Agent 的完整系统提示由多层注入组成，每层有独立的维护逻辑：

```
层 1：人格定义（稳定，很少变化）
    Alice 的核心身份、价值观、基本行为规范

层 2：能力边界（中等频率变化）
    可用工具的说明、工具使用的最佳实践

层 3：上下文注入（每次对话重新构建）
    项目记忆（ALICE.md）
    用户画像（identity / workflow / voice）
    当前激活的 Skills

层 4：动态追加（每轮迭代更新）
    当前日期时间
    本次会话的拒绝摘要
    渠道特定的前缀指令
```

**分层的好处**：
- 稳定的层（1、2）可以充分利用 LLM 的 KV Cache，降低计算成本
- 变化的层（3、4）独立维护，不影响稳定层的缓存命中率
- 每层有独立的负责人：层 1 由产品经理维护，层 3 由用户数据驱动，层 4 由代码自动生成

---

## KV Cache 优化：日期注入的位置

KV Cache 让 LLM 对之前见过的前缀不需要重新计算，大幅降低延迟和成本。

**日期注入的错误做法**：

```
// 放在系统提示开头
"当前时间：2026-04-20 10:30:15 (UTC+8)
 你是 Alice，一款 AI 助手..."
```

每次对话时间不同，系统提示前缀每次都变了，KV Cache 完全失效。

**正确做法**：

```
// 系统提示主体（稳定部分，KV Cache 命中）
"你是 Alice，一款 AI 助手..."

// 最后追加动态内容（不破坏前缀缓存）
"[Dynamic Context]
 Current time: 2026-04-20 10:30:15 (UTC+8)
 Permission denials this session: ..."
```

让稳定内容在前，动态内容追加在末尾，前缀 Cache 始终有效。

## [PROTECTED] / [MUTABLE] 分区

系统提示里的人格定义部分需要进一步细分：

```
[PROTECTED]
Alice 的核心身份。这部分永远不会被修改，无论用户请求还是 AI 自主决策。

你是 Alice，一款本地优先的个人 AI 助手...
（核心价值观、行为底线、安全规则）
[/PROTECTED]

[MUTABLE]
可以根据用户偏好和使用模式逐渐调整的行为规范。
（回答风格偏好、默认语言、常用领域知识等）
[/MUTABLE]
```

这个分区让 L0 自进化有了精确的操作对象：只能修改 `[MUTABLE]` 块，`[PROTECTED]` 块由代码级别的 guard 保护。

## 工具描述的写法原则

工具描述（`description` 字段）是提示词工程里最高密度的部分，每个字都在影响 LLM 的决策。

**四要素**：

1. **是什么**：一句话说清功能
2. **什么时候用**：触发条件要具体，避免模糊词
3. **什么时候不用**：明确排除场景（防止误用）
4. **输出说明**：告诉模型会拿到什么格式的结果

**写法示例**：

```
错误（只有"是什么"）：
  description: "搜索网络获取信息"

正确（四要素齐全）：
  description: |
    通过搜索引擎获取最新信息。

    适用场景：
    - 需要了解最新资讯或实时数据（如股价、新闻）
    - 知识截止日期之后发生的事件
    - 需要多个来源的信息做对比

    不适用：
    - 可以直接从本地文件或代码回答的问题
    - 用户已经提供了足够信息的情况

    返回：相关网页的标题、链接、内容摘要的列表
```

## 9 节上下文压缩格式

第五章提到了 Level-3 Collapse 压缩，需要 LLM 把早期对话压缩为结构化状态。

压缩提示词要求输出固定的 9 节格式：

```markdown
## 当前任务
（一句话描述：用户请求了什么）

## 工作目录
（当前的工作目录，以及对项目的了解）

## 进行中的工作
（正在处理什么，进度如何）

## 挂起的决策
（需要用户确认的决策，或需要更多信息才能继续的地方）

## 最近完成的工作
（最近几步完成了什么，防止重复执行）

## 发现的关键信息
（对完成任务有用的重要信息：路径、API、约定等）

## 遇到的问题
（已知障碍或错误，以及如何应对）

## 用户偏好
（用户在本次对话中表达的偏好或要求）

## 下一步行动
（明确的下一步，用于压缩后快速恢复状态）
```

**为什么是 9 节，而不是"生成摘要"？*

自由形式的摘要质量不稳定，有时会丢失关键信息。结构化的 9 节格式让模型有明确的"填写清单"，减少遗漏。这也让解压缩后的恢复更可靠：上下文重新注入时，格式一致，容易处理。

---

## 压缩提示词的特殊标记

压缩是一个特殊的 LLM 调用，需要模型**只输出文本**，不触发任何工具调用。

在压缩提示词开头加入：

```
你是一个上下文压缩器，职责是将对话历史压缩为结构化摘要。
在此任务中，你只能输出文字，不能调用任何工具。
```

同时，在代码里把这次 LLM 调用标记为 `querySource: 'compact'`，Agent 循环检测到这个标记后，即使压缩过程本身上下文很大，也不会递归触发再次压缩。

## 多 Agent 提示词指南

对 Coordinator Agent 的系统提示有特殊要求：

```
并行研究优先：
  "当任务涉及多个独立的信息收集子任务时，
   先并发收集，再综合分析，
   不要等第一个子任务完成后才启动第二个。"

禁止委托综合：
  "你的职责是综合所有子 Agent 的结果。
   不允许把综合工作交给子 Agent 完成，
   也不允许说'根据你的发现'，
   你是唯一看到全局的 Agent。"

结果验证：
  "对于关键决策，考虑启动验证子 Agent，
   让不同的 Agent 独立得出结论，
   如果结论一致则更可信。"
```

## Skill 的指令密度

Skill 的正文是最终被注入到系统提示里的内容，它的质量直接决定 AI 执行的质量。

**高质量 Skill 正文的特征**：

```
❌ 低密度：
  "帮用户写好文章"

✅ 高密度：
  "第一步：询问用户的目标受众和核心想表达的一个观点。
   等用户回复后进入第二步。

   第二步：用 web_search 工具搜索以下关键词：[用户主题] + site:zhuanlan.zhihu.com
   获取 3-5 篇同主题的高质量文章，分析其结构。

   第三步：输出大纲（Markdown 格式），每个章节包含：
   - 标题
   - 核心论点
   - 预计字数
   询问用户是否认可大纲，等待确认。

   第四步：逐章写作..."
```

每个步骤：说清楚做什么（操作）、用什么工具、输出什么格式、是否等待用户输入。
