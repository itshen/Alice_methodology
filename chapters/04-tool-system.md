# 第四章：工具系统

> 工具不是函数，是声明式契约。这个认知转变，是构建可靠 Agent 的关键一步。

---

## 设计动机：从函数到契约的认知转变

大多数开发者第一次做工具系统时，思路是"把函数包装成工具"。这个思路能跑起来，但很快会遇到一面墙：

**框架需要知道，但函数无法表达的信息：**

- 这个工具可以和其他工具并发执行吗？（并发安全性）
- 执行前需要用户确认吗？（权限模型）
- 子 Agent 可以调用这个工具吗？（工具可见性边界）
- 计划模式下应该怎么处理？（执行语义）
- 结果太大时应该怎么处理？（输出上限）

这些信息不是执行逻辑的一部分，是工具的**元数据**。把元数据和执行逻辑放在一起，就得到了"声明式工具契约"：工具不只告诉框架"我能做什么"，还告诉框架"框架应该怎么用我"。

---

## 业内工具系统设计对比

| 框架 | 工具定义方式 | 并发控制 | 权限模型 | 动态工具集 |
|------|-----------|---------|---------|---------|
| **LangChain** | Python 函数 + 装饰器 | 无内置机制 | 无内置机制 | 支持运行时注册 |
| **OpenAI Function Calling** | JSON Schema 声明 | 无内置机制 | 无内置机制 | 每次调用传入 |
| **Claude Tool Use** | JSON Schema + description | 无内置机制 | 无内置机制 | 每次调用传入 |
| **AutoGen** | Python 函数 + register_function | 无内置机制 | 无 | 运行时注册 |
| **Hermes** | Python 装饰器 + registry | 无 `isConcurrencySafe` | 无内置机制 | 运行时 AST 扫描发现 |
| **Alice** | TypeScript 对象 + 元数据字段 | `isConcurrencySafe`（含动态判断） | `requiresPermission` 字段 + 五层模式 | 动态组装（按 Skill/角色/设置过滤） |

**各框架的共同缺陷：**

大多数框架把工具定义成函数，把并发控制、权限判断留给上层代码处理。这导致框架本身无法做出智能的编排决策，只能选择：全部串行（安全但慢）或全部并行（快但不安全）。

**Alice 的关键设计：`isConcurrencySafe` 可以是函数**

大多数工具的并发安全性是固定的（搜索永远安全，写文件永远不安全）。但有些工具的安全性取决于输入：比如"写入到不同文件"可以并发，"写入到同一文件"不能。允许 `isConcurrencySafe` 是一个根据输入动态计算的函数，让框架能做出更精细的编排决策。

---

## 为什么不能只是函数

最简单的工具系统是这样的：

```
tools = {
  "search": lambda query: do_search(query),
  "write_file": lambda path, content: do_write(path, content),
}
```

当工具数量少、场景简单时，这没问题。但当你开始面对这些问题，就会发现纯函数不够用：

- 这个工具可以和其他工具并发执行吗？
- 执行前需要用户确认吗？
- 结果太大时应该怎么处理？
- 子 Agent 可以调用这个工具吗？
- 计划模式下（只规划不执行）应该怎么处理？
- 这个工具是只读的还是会修改状态？

这些问题，都需要工具"告诉框架"，而不是由框架猜测或硬编码。

**工具是声明式契约**：它不只是执行代码，还包含关于自身行为的元数据声明，让框架可以做出正确的编排决策。

---

## 工具声明的关键字段

一个完整的工具声明应该包含：

```
name            工具唯一名称（LLM 用它识别工具）
description     功能描述（直接影响 LLM 的调用决策！）
inputSchema     输入格式（用于类型验证和 JSON Schema 生成）

isReadOnly      是否只读（不修改任何状态）
isDestructive   是否有不可逆影响（删除文件、发送消息等）
isConcurrencySafe  是否可以与其他工具并行执行
                    → 可以是固定的 true/false
                    → 也可以是函数：根据具体输入动态判断

requiresPermission  执行前是否需要权限检查
maxResultSizeChars  输出大小上限（超出则写文件返回路径）

execute(input, ctx) → ToolResult  实际执行函数
```

---

## `description` 是最重要的字段

工程师常常低估 `description` 的重要性。实际上，**工具描述的质量直接决定 LLM 是否会在正确的时机调用它**。

**反例**（太简单）：
```
description: "写入记忆"
```

**正例**（包含触发时机、使用场景、不适用场景）：
```
description: |
  将重要信息持久化到项目记忆文件。

  应该使用的场景：
  - 用户明确要求"记住"某件事
  - 发现了项目的特殊约定或规则
  - 完成了重要任务，需要记录结果供未来参考

  不应该使用的场景：
  - 临时性信息（这次对话用完就不需要了）
  - 可以从代码文件直接查到的信息
  - 会频繁变化的动态数据
```

写 description 的心态：把它当成**给模型的使用说明书**，而不是给人类开发者看的注释。

## 并发调度：`isConcurrencySafe` 的价值

当一轮对话里 LLM 发起多个工具调用时，应该串行还是并行？

串行：安全，但慢。五个工具调用如果每个都需要 2 秒，就是 10 秒等待。

并行：快，但可能产生竞态。如果两个工具都写同一个文件，结果不可预测。

`isConcurrencySafe` 提供了中间路径：

```
调度逻辑：

把工具调用列表分成批次：
  - 连续的"并发安全"工具合为一批，并行执行
  - "不安全"的工具单独一批，串行执行

例子：
  [file_read A, web_search B, file_write C, file_read D]
       ↓ 分批
  batch 1: [file_read A, web_search B]  → Promise.all()
  batch 2: [file_write C]               → 单独执行（写操作）
  batch 3: [file_read D]                → 单独执行
```

`isConcurrencySafe` 还可以是**函数**，根据具体输入动态决定。最典型的例子是 `agent` 工具（触发子 Agent）：如果子任务是只读的，可以并行；如果要修改文件，需要串行。

## 执行上下文注入（ToolContext）

工具执行时，除了用户提供的 `input`，还会注入一个执行上下文：

```
workdir          当前工作目录
sessionId        会话标识
requestPermission  权限申请回调（工具需要额外权限时调用）
abortSignal      中止信号（用户取消时接收）
getKey(provider) 获取指定服务商的 API Key
getSettings()    获取当前设置
emitEvent(event) 发射事件（用于子 Agent 进度上报）
渠道上下文        微信/Telegram 相关信息
```

**关键设计**：工具代码不依赖任何全局变量，所有需要的东西通过 `ctx` 注入。这让工具可以在任何环境里运行（主进程、Worker Thread、测试环境），也让工具的依赖关系清晰可见。

## 大结果落盘

某些工具（比如 `bash` 执行 `find /` 或 `grep` 大文件）可能返回几十 MB 的内容。把这些内容直接放进 LLM 上下文是灾难性的。

处理方案：

```
执行工具
    ↓
检查结果大小是否超过 maxResultSizeChars
    ↓ 超过
写入临时文件
    ↓
返回："结果已写入 /tmp/result_xxx.txt，使用文件读取工具查看"
    ↓ 未超过
直接返回结果
```

两个注意点：
1. **优先写入工作目录**（方便 AI 后续读取），工作目录不可用时才用临时目录
2. **注册进程退出清理**：临时文件在进程退出时自动删除，不留垃圾

## 子 Agent 的工具限制

子 Agent 不应该拥有与父 Agent 相同的完整工具集。需要禁止的工具包括：

- `agent`：禁止子 Agent 递归创建孙子 Agent（防止失控）
- `ask_user`：子 Agent 不能直接弹窗问用户（权限应该通过父 Agent 传递）
- 计划模式控制工具：这是会话级别的设置，子 Agent 不应该改变它
- 渠道发送工具：发送消息由主 Agent 统一负责
- 记忆写入工具：子任务的中间状态不应该写入长期记忆

**MCP 工具是例外**：子 Agent 需要同样的外部工具能力，MCP 工具始终保留。

---

## 工具的类型分类

按功能分类，工具通常包含以下几类：

| 类别 | 例子 | 特点 |
|------|------|------|
| 文件系统 | 读文件、写文件、搜索 | 只读工具并发安全 |
| 网络 | 搜索、抓取网页 | 并发安全，但注意频率限制 |
| 系统执行 | bash、运行脚本 | 非并发安全，需要权限 |
| 记忆管理 | 读/写项目记忆、向量搜索 | 写操作非并发安全 |
| 外部 API | 天气、股价、日历 | 通常并发安全 |
| AI 能力 | 图像生成、子 Agent | 根据场景决定 |
| 交互 | 询问用户 | 特殊处理（需要等待用户） |
| 自进化 | 修改 UI、生成页面 | 高权限，需要严格限制 |
