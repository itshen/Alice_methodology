# 第七章：权限系统

> "默认保守，显式放行。"这是权限设计的最小原则，但落地的方式千差万别。

---

## 设计动机：为什么权限系统如此复杂

Agent 的权限问题，本质上是一个**不可能三角**：

```
       全自动化（省心）
          △
         / \
        /   \
       /     \
安全性 ────── 可配置性
```

三个角没有办法同时最大化：
- 要完全安全，就必须每个操作都问用户，但用户体验极差
- 要完全自动，就必须放开权限，但误操作和安全风险剧增
- 要完全可配置，规则系统会变得极其复杂，大多数用户根本不会用

传统软件的解法是"固定权限列表"（读、写、执行），但 LLM Agent 的操作是语义化的，"帮我清理项目"可能需要删文件，"帮我重构代码"可能需要修改几十个文件，没有办法用固定列表预先定义。

---

## 业内权限设计方案对比

| 方案 | 代表实现 | 核心机制 | 优点 | 局限 |
|------|---------|---------|------|------|
| **全信任** | 大多数早期 Agent 框架 | 工具调用直接执行 | 实现简单，效率高 | 任何工具调用错误都直接执行 |
| **人机确认** | Claude Code 默认模式 | 危险操作弹窗让用户确认 | 安全，用户可控 | 频繁打断，影响效率 |
| **规则白名单** | Cursor（Ask Mode/Agent Mode） | 预设工具集白名单 | 可预期，安全 | 灵活性差；难以处理边界情况 |
| **沙盒隔离** | Docker 容器化 Agent | 在容器内执行，限制系统访问 | 强隔离 | 沙盒配置复杂；性能开销大 |
| **LLM 分类器** | 部分企业方案 | 用小模型评估操作风险 | 语义级安全判断 | 分类器本身可能误判；增加延迟 |
| **分层五模式** | Alice | 五种模式 + 规则匹配 + LLM 分类器 + 否决跟踪 | 覆盖完整需求谱；可精确配置 | 实现最复杂 |

**各方案的根本问题：**

"全信任"在生产环境根本不可用。"纯人机确认"会让用户在长任务中疯掉：一个 50 轮的代码任务，可能要确认 30 次。"纯规则白名单"适合可预期的工作流，但 Agent 的价值恰恰在于处理不可预期的情况。

Alice 的答案是：**让用户自己选择信任等级**，而不是替用户决定。五种模式让不同场景都有最合适的配置：CI 环境用 `bypass_permissions`，日常用 `default`，高风险操作前切换到 `plan`。

---

## 权限的根本矛盾

Agent 越自动化，用户越省心；自动化程度越高，风险也越大。

`rm -rf ~/important-files` 可能在某个逻辑链下看起来"合理"，但执行是不可逆的。

权限系统要解决的核心问题是：**在用户期望的自动化程度和实际操作风险之间，找到正确的平衡点，并让这个平衡点可以被精确配置。**

---

## 五种权限模式

从最保守到最激进，覆盖完整的需求谱：

| 模式 | 含义 |
|------|------|
| `plan` | 只规划，不执行。所有工具调用都被阻断，Agent 只能输出文字 |
| `default` | 危险操作需要用户弹窗确认 |
| `accept_edits` | 自动接受文件编辑，bash 等命令仍需确认 |
| `dont_ask` | 全自动，但保留 AI 分类器兜底（智能判断危险操作） |
| `bypass_permissions` | 完全绕过权限检查，适合 CI/自动化场景 |

**`plan` 模式**：一个非常有用但常被低估的功能。"告诉我你打算做什么，但先不要做"。对于不确定的任务，先看计划，觉得没问题再执行。

**`dont_ask` 模式**：本质是"不问用户，但 AI 会自己判断"。对于明确危险的操作，AI 分类器拒绝；对于明确安全的操作，自动放行；对于不确定的，走轻量规则。

---

## 决策流程：责任链

```
输入：工具名 + 工具输入参数
    ↓
Mode 判断
    plan → 只读工具放行，其他全阻断
    bypass → 全放行
    accept_edits → 文件编辑类放行，其他继续
    ↓
Session 记忆
    这次对话里用户已经允许过的操作 → 放行
    ↓
规则匹配
    匹配到 allow 规则 → 放行
    匹配到 deny 规则 → 拒绝
    匹配到 ask 规则 → 继续往下
    ↓
dont_ask 模式的 AI 分类器
    → 发起一次轻量 LLM 调用判断是否安全
    → 安全 → 放行
    → 危险 → 拒绝
    ↓
默认模式的用户确认弹窗
    → 用户点击允许 → 放行，并记入 Session 记忆
    → 用户点击拒绝 → 拒绝，并记入拒绝追踪
```

每个决策都携带 `reason` 字段（`mode_bypass` / `rule_allow` / `ai_safe` / `user_allow` 等），便于后续审计和调试。

## 规则系统

权限规则是用户配置具体放行/禁止条件的地方：

```yaml
permissions:
  # 允许在工作目录下自由编辑文件
  - action: allow
    tool: file_write
    pathPattern: "~/Projects/**"
  
  # 禁止 git push（防意外推送）
  - action: deny
    tool: bash
    pattern: "git push"
  
  # npm install 需要确认（可能很慢）
  - action: ask
    tool: bash
    pattern: "npm install"
```

规则的三个维度：
- **action**：allow（放行）/ deny（拒绝）/ ask（询问用户）
- **tool**：工具名，`*` 匹配所有工具
- **pattern / pathPattern**：更精细的匹配（命令前缀 / 文件路径 glob）

规则按顺序匹配，第一个命中的规则生效。

---

## 拒绝追踪：防止 AI 反复碰壁

当某个操作被拒绝，如果不告知 AI，AI 可能在下一轮循环再次尝试同一个操作，反复碰壁。

解决方案：维护一个"本次会话拒绝记录"，在每轮迭代开始时，把最近的拒绝摘要追加到 system 消息：

```
注意：以下操作在本次会话中已被拒绝：
- bash: git push origin main
- bash: rm -rf /tmp/cache

请不要再次尝试这些操作。
```

这个摘要追加在 system 消息末尾（不是开头），对 KV Cache 友好。

## `~/.alice` 目录的特殊规则

自进化功能（修改 Skill、更新系统提示、生成 Widget）需要频繁写入 `~/.alice/` 目录。如果每次都需要用户确认，体验会非常差。

解决方案：**对 `~/.alice/` 目录的写操作自动放行**。这是 Alice "自己的家"，改变自己不需要每次经过用户审批。

但要注意边界：`~/.alice/` 的自动放行只适用于 Alice 的进化操作，不应该扩展到整个用户目录。

这体现了一个更大的原则：**"改变自己"和"改变用户系统"需要不同的权限级别。**

---

## AI 分类器的工程权衡

`dont_ask` 模式中，AI 分类器需要在速度和准确性之间权衡：

**启发式规则先行**：明确安全的操作（读文件、搜索网络）和明确危险的操作（`rm -rf`、`git push --force`）不走 LLM 调用，直接本地判断。

**缓存**：相同的工具+输入组合，缓存分类结果（TTL 约 5 分钟）。

**降级**：如果分类器连续多次拒绝某个操作（AI 认为危险），说明可能遇到了分类器盲区，降级为人工确认，而不是无限拒绝让任务卡死。

---

## 权限系统的核心原则

1. **默认保守**：没有显式放行的危险操作，需要确认
2. **明确可配置**：五种模式覆盖完整需求谱，用户可精确控制
3. **分层防护**：任何一层拦住都算成功
4. **决策可审计**：每个决策带 `reason`，可以追溯为什么这样决定
5. **失败安全**：分类器不确定时，降级到人工确认，而不是自动放行
